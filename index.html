▸ THE PROBLEM WITH SINGLE-RUN BENCHMARKS

Single-run, single-prompt benchmarks are misleading. A model can appear slow on a generic prompt yet be excellent in its specific domain. Ollama's KV-cache means the second run on the same prompt is artificially faster than the first. And comparing a reasoning model (optimized for quality) directly against a general model (optimized for speed) produces unfair results.

OllamaVision solves this by running NUM_RUNS=3 passes with different prompts per run, comparing each model only against models in the same category, and producing a multi-dimensional score that accounts for speed, energy efficiency, thermal behaviour, RAM pressure, result stability, and load time.

▸ COMPLETE FEATURE SET
🔁
Multi-Run with Prompt Rotation
3 measured runs per model, each using a different prompt from a curated set. Eliminates KV-cache bias. 1 warmup run precedes measurement to stabilize GPU clock speeds.
📊
Category-Scoped Scoring
Models are ranked by TPS percentile within their own category — GENERAL, CODE, MATH, REASONING, VISION, EMBEDDING. A slow reasoning model can still score highly if it outperforms other reasoning models.
🎯
6-Factor Efficiency Score
Weighted score 0–100 combining speed (35%), energy efficiency (25%), thermal penalty (15%), RAM penalty (10%), inter-run stability (10%), and model load speed (5%).
⚡
Streaming TTFT Measurement
Default streaming mode (stream=true) measures Time-To-First-Token precisely. TTFT is stored in-memory and exported to the JSON summary. Falls back to blocking mode on timeout.
🌐
Cyberpunk Interactive Dashboard
Single self-contained HTML file with 16 Plotly charts across 7 tabs. No server required — open directly in any browser. IBM Plex Mono + Orbitron typography.
🖥️
Real-Time Terminal Dashboard
Rich Live 6-panel terminal layout refreshed at 2 Hz during benchmark. Shows progress, live hardware telemetry, running stats, and a scrollable results table.
🔋
Per-Rail Power Monitoring
Reads Jetson INA3221 sensors: VDD_GPU_SOC, VDD_CPU_CV, VIN_SYS_5V0. Computes tokens per watt for each model. Hardware window is averaged over the actual inference interval.
💾
Atomic Checkpoint / Resume
Checkpoint writes go through .tmp → rename atomically. If the process is killed mid-benchmark, just re-run — it picks up exactly where it left off.
🔍
Dynamic Model Discovery
Discovers all installed models via /api/tags with fallback to ollama list. No hardcoded model lists. Category is auto-detected from name heuristics or overridden in config.
🧊
Adaptive Cooldown
Before each model, checks the last 30 seconds of GPU temperature history. Only waits if the GPU is above the cooldown threshold. Skipped entirely if already cool.
🔬
Simulation Mode
Full pipeline runs on any Linux machine with --simulate. Generates realistic synthetic telemetry (Gaussian noise over Jetson AGX Orin baselines) so the entire system can be developed and tested without hardware.
🎛️
Targeted Benchmarking
Filter by category (--filter CODE,MATH) or by model name substring (--models qwen,phi4). Override runs and timeout per invocation without touching config.

        └── _recalculate_scores_final()   ← recompute percentiles · full dataset

              ├── _export_json_summary()      ← TTFT · per-model hw averages · filters

▸ EFFICIENCY SCORE FORMULA
SCORE = (0–100 for each factor, then weighted sum)
+ Speed — TPS percentile within category
35%
+ Energy — tokens per watt (ref: 4 tok/W)
25%
− Thermal penalty — GPU temp (ok below 62°C)
15%
− RAM penalty — RAM usage (ok below 12 GB)
10%
+ Stability — 1/CV inter-run (CV ≤ 3% → max)
10%
+ Load speed — model load from disk (ok <30s)
5%

📋 RECOMMENDATION DECISION TREE
category == EMBEDDING?    → EMBEDDING   (TPS doesn't apply)
benchmark failed?         → ERROR        (retry or check Ollama)
score ≥ 68?               → KEEP★        (excellent — deploy with confidence)
score ≥ 42?               → KEEP         (good — worth keeping)
score ≥ 22?               → OPTIONAL     (ok if use case justifies it)
TPS < cat_min AND RAM > 28GB? → REMOVE   (slow + memory-hungry → ollama rm)
otherwise                 → REVIEW       (below threshold — evaluate manually)

Note: REVIEW does not mean the model is bad — only that it was slower than expected for its category under benchmark conditions. It may excel in real usage where quality matters more than raw speed.

▸ DASHBOARD TAB GUIDE
[ OVERVIEW ] — at a glance

6 KPI cards (fastest, best score, most efficient, avg power, avg GPU temp, keep/remove counts). TPS bar with ±1σ error bars. Recommendation donut. TPS vs RAM scatter (dot size = power). Efficiency score horizontal bar.

[ PERFORMANCE ] — speed analysis

Full-width TPS chart. Efficiency score + tokens/watt. Latency breakdown (model load vs inference stacked bar). Load-time vs inference-time scatter — detects models that don't fit in VRAM.

[ BY CATEGORY ] — strategic view

6 sub-charts: avg TPS, max TPS, avg score, tok/W, avg GPU temp, avg power — all by category. Best-model-per-category table. Multidimensional radar chart (speed · efficiency · thermal · GPU% · eco).

[ THERMAL & POWER ] — hardware

Per-sensor temperatures (GPU/CPU/SOC/TJ) with 75°C and 85°C warning lines. Stacked power-rail bar (GPU+SOC / CPU+CV / SYS5V0). Dual-axis GPU load vs frequency. RAM usage heatmap with 28 GB limit line.

[ STABILITY ] — multi-run stats

CV% bar per model with STABLE/GOOD/VARIABLE/UNSTABLE threshold lines. Box-plot of individual run TPS distributions per model — a wide box means inconsistent results across the 3 runs.

[ DECISIONS ] — fleet management

Decision matrix scatter (TPS vs Score, colored by recommendation). REVIEW/REMOVE candidates table with decision reasons. Error log. Embedding model table.

▸ INSTALLATION
1 · CLONE & INSTALL DEPENDENCIES
$ git clone https://github.com/sergio-usma/ollama-vision-benchmark-suite
$ cd ollama_benchmark
$ pip install -r requirements.txt --break-system-packages
# On Jetson with system Python use --break-system-packages
2 · (OPTIONAL) JETSON HARDWARE MONITORING
$ sudo pip install -U jetson-stats --break-system-packages
# Requires restart after install
# Without jtop, simulation mode activates automatically
3 · (OPTIONAL) ADD A VISION TEST IMAGE
$ cp your_image.jpg assets/test_image.jpg
# Required for VISION model benchmarking
4 · VERIFY ENVIRONMENT
$ python3 main.py --check
